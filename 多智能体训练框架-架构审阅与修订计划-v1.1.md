# 架构审阅与修订计划 v1.1

## Context

仓库当前已实现"多智能体自动化训练框架架构设计v1.0.md" + "CLI引擎桥接适配升STDIO Streaming JSON-RPC 通信层.md"两份设计文档中的一部分：**Engine桥接层（agents/engines/*）+ BaseAgent基础设施 + 全部6个Planning Agent**，并配有对应单元测试。`task-breakdown-c877cf.md`定义的T0-T8任务中，实际只完成了T0/T1/T2三个阶段；T3（训练脚本，部分完成但有bug）、T4（Execution Agents/log_adapters/data_format_converters）、T5（Orchestrator闭环）、T6（Knowledge）、T7（CLI入口/文档）、T8（端到端回归）均未开始。

本次审阅目的：在继续推进T3-T8之前，先系统审查已完成部分（尤其是刚做完协议升级的Engine桥接层）是否存在正确性问题，并核实设计文档中反复强调的"需实现阶段实测确认"的风险点是否真的被验证过。审阅采用2个并行Explore agent分别深读Engine层与Planning层代码，并由本人交叉核对了关键文件（jsonrpc_transport.py/process_manager.py/codex_engine.py/claude_engine.py/opencode_engine.py/base_agent.py/schemas.py/scripts/*.sh）。

结论：**Engine桥接层存在多个真实的并发/资源管理bug，且设计文档反复强调的"必须真实调用CLI二进制验证协议细节"这一验收标准完全没有被执行**——所有测试都只跑在自写的fake协议server上，三个engine代码头部注释里的"实测结论"在仓库里找不到任何可复现的证据支撑。这是当前最大的风险，必须在继续搭建Execution/Orchestrator之前处理，否则整个训练闭环会建立在未经验证的协议假设之上。

## 一、审阅发现

### 1.1 Engine桥接层（agents/engines/*）—— 高优先级问题

| # | 文件:行 | 问题 | 影响 |
|---|---|---|---|
| P0-1 | `process_manager.py:67-72` | `Popen(..., stderr=subprocess.PIPE)`，但全仓库没有任何代码读取`proc.stderr` | CLI进程往stderr写多了（verbose模式很常见）会导致管道缓冲区写满、子进程阻塞，进而卡死整个turn（父进程还在等stdout响应）。这是真实的死锁风险，测试用的fake server从不写stderr，完全没被覆盖 |
| P0-2 | `process_manager.py:73` + `:144-148` | `start()`内`self._restart_attempts = 0`无条件重置，而watchdog触发的重启本身也调用`start()`，导致每次重启成功后计数器立刻被清零 | `RestartPolicy.max_attempts`对"进程能正常拉起但立刻退出"这种最常见的crash-loop场景**完全不生效**，会无限重启。现有测试`test_auto_restart_on_abnormal_exit`用"等到exit_codes长度=3就停止观察"的方式验证，并未真正验证重启会停止，掩盖了此bug |
| P0-3 | `jsonrpc_transport.py:136-143` (`_write`) | `request()`/`notify()`调用的`_write()`没有任何锁保护stdin写入 | 若同一engine实例被多线程并发调用（如`run()`与`cancel()`分属两个线程同时调用transport.request），两次write可能交错，产生损坏的NDJSON行，被`_dispatch_line`的`except JSONDecodeError: return`静默丢弃且无日志 |
| P0-4 | `codex_engine.py:147-152`、`opencode_engine.py:115` | 每次`run()`都调用`transport.on_notification(method, handler)`注册新闭包，但`JsonRpcTransport.on_notification`只会`append`，从不注销 | Monitor Agent按设计需要对同一engine实例反复调用`run()`（每`interval_minutes`一次，持续数小时/数天）——每调一次就多一个handler常驻，旧闭包会持续接收后续turn的通知并往自己已经"过期"的`text_parts`/`events`里追加、并重新调用调用方传入的`on_event`回调。结果：内存无界增长 + 通知分发耗时线性增长 + 若`on_event`绑定了某次turn专属的资源（如日志文件句柄），会在下一turn被错误地再次触发。opencode_engine里虽然有`sessionId`过滤，但由于session在多次run()间是复用的（同一session_id），这个过滤并不能阻止跨turn泄漏，反而制造了"已经过滤过"的假象 |
| P0-5 | `claude_engine.py:223-230` (`cancel()`) | `proc.send_signal(signal.SIGINT)`，异常被`except Exception: pass`吞掉 | 当前开发环境是Windows（`win32`）。CPython在Windows下`Popen.send_signal()`只支持`SIGTERM`/`CTRL_C_EVENT`/`CTRL_BREAK_EVENT`，传入`SIGINT`会抛`ValueError: Unsupported signal: 2`，被静默吞掉——`cancel()`在Windows上**永远是空操作**，且没有任何测试覆盖`cancel()` |
| P0-6 | `claude_engine.py:138-141` (`_dispatch`的`assistant`分支) | `if text and not self._current_text_parts:`——只在`text_parts`当前为空时才追加 | 由于engine启动命令固定带`--include-partial-messages`（`_build_command`），delta事件几乎总会先于完整assistant消息到达并填充`text_parts`，导致这段代码在当前配置下基本是死代码；目前靠`result.result`字段兜底才没出问题，但这是脆弱的隐性依赖，一旦上游协议细节变化就会静默丢文本 |
| P0-7 | `agents/base_agent.py:32-37` | `_ENGINE_OUTPUT_ERRORS`把三个engine各自的"schema校验失败"和"turn执行出错(如codex的`error`通知、opencode的`JsonRpcError`)"统一映射成同一个异常类型 | `BaseAgent.run()`的重试逻辑无法区分"LLM输出格式不对，该给纠错反馈"和"底层CLI/沙箱/网络出错，纠错反馈完全无意义"，会在infra错误上浪费重试预算并给出文不对题的"请修正JSON"提示 |
| P0-8 | 全仓库 | 没有任何代码/脚本实际调用真实的`codex`/`claude`/`opencode`二进制 | `codex_engine.py`/`claude_engine.py`/`opencode_engine.py`头部注释都写着"实测结论（当前环境 xxx 版本）"，但仓库里没有任何可复现的验证产物（脚本输出、日志、测试）能证明这些协议细节（方法名/参数结构/事件名）真的针对当前环境的真实CLI跑通过。`test_engines_consistency.py`及三个`tests/fakes/*`只是自己写的模拟server，只验证了"Python代码与自己写的假设一致"，并不能验证"假设本身是否正确"。这直接违反了`task-breakdown-c877cf.md`里T1-4/T1-5/T1-6明确写的验收标准（"真实调用一次claude验证…"等） |
| P0-9 | `opencode_engine.py`（整个文件） | 只实现了ACP路径，没有设计文档明确要求的"ACP不可用时降级为`opencode run`一次性subprocess"路径 | 若当前`opencode`版本ACP不稳定/不可用，三个engine能力就不对等且无兜底，而设计文档T1-6把这个降级路径列为必须交付项 |

其他非阻塞但值得记录的问题：
- `codex_engine.py`：`_on_turn_started`捕获的`current_turn_id`从未被读取（死代码）；`cancel()`发送`turn/interrupt`的参数形状从未针对真实CLI验证。
- `jsonrpc_transport.py`：`stop()`不会等待reader线程真正退出（daemon线程，不阻塞进程退出，但调用方以为`stop()`返回即完全清理干净）；`_handle_notification`吞掉handler异常且全模块没有`logging`，出问题很难排查；`include_jsonrpc_field=False`分支从未被任何调用方使用过，是死代码。
- `configs/agents.yaml`：设计文档提到的`engines.process_pool: per_call|warm`全局开关未实现，当前实际行为是"每个Agent实例持有一个长驻engine贯穿其生命周期"（偏warm），但没有配置项可选。
- `PipelineStage.engine`（`agents/planning/schemas.py:75`）是自由文本`str`，未约束为`configs/training_engines.yaml`中已注册的5个引擎名之一，LLM产出"Trl"这类大小写不一致的值会在未来Trainer Agent按引擎名查表时失配。

### 1.2 Planning Agents层 —— 整体质量较好，小缺口

- 六个Planning Agent + `plan_reviewer_agent.py` + `report_writer_agent.py`都是规范的`BaseAgent`子类，`schemas.py`里的pydantic模型与设计文档描述的字段对得上（含`citations`引用要求、`ReviewIssue.category`三分类与分层回退优先级一致）。
- `training_plan_generator.py`里的markdown模板与设计文档要求的8个章节标题逐字一致，是忠实转录。
- **`plan_reviewer_agent.py`的分层回退目前只是一个纯函数`determine_rollback_target(issues)`**，只做"给定问题列表判断该回退到哪个Agent"，没有任何代码真正执行"重新调用被回退的Agent→重新评审→直到通过或达到最大重试次数后升级人工介入"这个闭环（设计文档190/281行明确要求，建议3次上限）——因为`orchestrator/state_machine.py`根本不存在，这段逻辑目前是"孤儿函数"，只在单元测试里被手工喂数据调用。
- `TrainingPlanOutput.markdown`字段没有任何结构性校验去确认其内容真的包含7个必需章节、且与并行的结构化字段（`pipeline_stages`/`data_format`）保持一致——完全依赖prompt层面的自觉，schema层面没有交叉校验。
- `training_plan_generator.py`的`_load_similar_cases_summary()`会读`knowledge_base/index.json`，但该文件目前不存在（`knowledge_base/`下只有`.gitkeep`），优雅降级为"暂无历史案例"——设计文档强调的"优先复用knowledge_base历史案例"这条决策优先级目前永远是空的，属于预期内的阶段性缺口（等T6完成后才会有内容）。
- 测试质量：`test_base_agent.py`对重试/幂等/异常耗尽路径覆盖扎实；`test_planning_agents.py`更多是"六个Agent能串起来跑通"的wiring测试，缺少真实schema的校验失败路径测试，也没有真正端到端跑一次"拒绝→回退→重新生成→再评审→通过"的闭环（因为没有调用方）。

### 1.4 训练脚本（scripts/*.sh）—— 一处疑似参数顺序错误

`scripts/llamafactory_run.sh:46-49`：
```bash
llamafactory-cli train \
  --output_dir "$RUN_DIR" \
  "$CONFIG_PATH" \
  2>&1 | tee "$LOG_FILE"
```
LLaMA-Factory CLI的标准调用方式是`llamafactory-cli train <config.yaml> [key=value ...]`（第一个位置参数是yaml路径，覆盖项用`key=value`而非`--flag value`）。当前脚本把`--output_dir`放在yaml路径之前，很可能无法被正确解析（未实测，需要在T3阶段用真实`llamafactory-cli --help`核实后修正，`trl_run.sh`的`--config/--output_dir`用法相对更接近trl CLI实际支持的形式，风险较低）。

## 二、修订后的交付计划

在原`task-breakdown-c877cf.md`的T0-T8基础上，**新增一个T1.5阶段**专门修复上述Engine桥接层问题并补齐真实CLI验证，**必须在启动T4（Execution Agents）之前完成**——因为Trainer/Monitor都是长驻调用engine.run()的重度使用者，会直接放大P0-1/P0-2/P0-4这几个bug的影响面。其余阶段维持原有依赖顺序，仅按审阅发现调整个别任务描述。

### T1.5 Engine桥接层修复与真实CLI验证（新增，阻塞T4）
- **T1.5-1** 修复`process_manager.py`：起一个独立线程持续drain `proc.stderr`（写入bounded buffer，供异常时附加到错误信息里，避免OOM）；修正`_restart_attempts`重置逻辑（只应在"距上次重启已稳定运行一段时间"或"用户主动调用start()"时清零，不能在watchdog触发的重启成功后立即清零）。
- **T1.5-2** `jsonrpc_transport.py`增加写锁包住`_write()`；补充并发写入的回归测试（两个线程同时`request()`验证不产生损坏帧）。
- **T1.5-3** 重构`codex_engine.py`/`opencode_engine.py`的通知处理：改为每次`run()`用**一次性/turn-scoped**的方式订阅（如`on_notification`返回一个可调用的注销句柄，`run()`结束时finally里注销），或者在`JsonRpcTransport`里加`off_notification()`并在`codex_engine.py`/`opencode_engine.py`的`run()`用`try/finally`确保注销。补充长时间多次`run()`调用后handler列表不增长的回归测试。
- **T1.5-4** `claude_engine.py`：`cancel()`改为跨平台安全实现（Windows下不再尝试`SIGINT`，退化为记录"当前环境不支持turn级取消，如需终止请调用stop()"，或使用`CREATE_NEW_PROCESS_GROUP`+`CTRL_BREAK_EVENT`方案，需实测选择）；修掉`_dispatch`里`assistant`分支的死代码判断（改为在有`_current_result_text`时优先使用它、否则始终追加assistant文本，不再用"当前是否为空"做门槛）。
- **T1.5-5** 引入区分"结构化输出校验失败"与"引擎/传输层错误"的异常分层（如给每个engine拆出`XxxOutputValidationError`和`XxxTransportError`两类），`base_agent.py`的重试逻辑只对前者追加纠错反馈，后者应直接向上抛出或采用不同的重试策略。
- **T1.5-6** `opencode_engine.py`补齐ACP不可用时降级为`opencode run`一次性subprocess的路径，并在返回结果/日志中标记降级状态，补充降级路径的测试。
- **T1.5-7**（关键，阻塞后续）**真实CLI二进制验证**：对`codex app-server`、`claude -p --output-format stream-json ...`、`opencode acp`分别执行一次真实调用（非fake），核实：方法名/参数结构（尤其`approvalPolicy`/`sandboxPolicy`形状、ACP的`session/*`方法名）、错误事件结构、`cancel()`对应的真实方法在当前版本是否存在。把验证过程和实际抓到的原始协议消息样本存档（如`docs/protocol_verification/*.jsonl`或类似），三个engine模块头部的"实测结论"注释改为引用该存档，不再是无出处的断言。若某协议特性在当前版本不可用，按设计文档要求的降级方案实现并明确标注。
- **T1.5-8** 补`.gitignore`（`__pycache__/`、`*.pyc`、`.pytest_cache/`等），并把已提交的`__pycache__`产物从git中移除。
- **T1.5-9** 修正`scripts/llamafactory_run.sh`的CLI调用参数顺序/形式（实测`llamafactory-cli train --help`后修正）。

### T3（延续原编号）训练脚本 —— 补充验证

- T8端到端回归新增一项：至少完整跑一次**真实CLI引擎**（非fake）的Planning阶段全流程，验证T1.5-7的验证产物在真实端到端场景下依然成立。
